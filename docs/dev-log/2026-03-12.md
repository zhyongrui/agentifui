  - persisted failed-login counters and lockout timestamps

## Gateway Auth Persistence

Added DB-backed service implementations in `apps/gateway`:

- `persistent-auth-service.ts`
- `persistent-audit-service.ts`

The default `buildApp` path now:

- loads `DATABASE_URL`
- runs migrations on startup
- ensures the default tenant exists
- uses persistent auth/audit services when database config is available
- falls back to the existing in-memory service only when no database is configured or tests explicitly inject doubles

Persisted state now includes:

- users
- password identities
- SSO identities
- invitations
- MFA factors
- MFA setup challenges
- MFA login tickets
- audit events
- sessions

Behavioral changes completed in this round:

- logout now revokes the session instead of only returning a success response
- sessions survive app restarts
- MFA enrollment survives app restarts
- invitation consumption survives app restarts
- audit events survive app restarts

## Public Preview Hotfix

Follow-up issue discovered during browser validation:

- the public web preview was reachable
- the gateway was reachable
- but browser-side auth requests still attempted to reach the gateway directly instead of going through a same-origin web endpoint

This caused the public login page to surface:

- `Unable to reach the auth gateway. Check the gateway server and try again.`

Hotfix implemented:

- added `apps/web/src/app/api/gateway/[...path]/route.ts`
- changed browser auth/workspace clients to call `/api/gateway/*`
- added `GATEWAY_INTERNAL_URL` support for the web server-side proxy target
- added web client tests for the same-origin proxy path
- changed persistent DB integration tests to `describe.sequential` and increased timeout so the full test suite can run without shared-database contention
