## P2-A6 Structured Conversation History Filters

Completed `P2-A6` by turning `/chat` history search into a structured filter surface instead of a plain text query box.

Implemented in this round:

- shared contracts now include:
  - history filter query types for `status`, `attachment`, `feedback`, and `tag`
  - `WorkspaceConversationListItem.attachmentCount`
  - `WorkspaceConversationListItem.feedbackSummary`
- gateway and services now include:
  - `GET /workspace/conversations` parsing and validation for `status`, `attachment`, `feedback`, and `tag`
  - in-memory and persistent history filtering against app tags, attachment presence, assistant feedback, and archived/active state
  - persistent history now applies structured filters after the DB read and only then slices the final page limit
- web history UI now includes:
  - `/chat` filter controls for `Status`, `Tag`, `Attachments`, and `Feedback`
  - active-filter summary chips
  - conversation cards that display app tags, attachment counts, and feedback summaries
- test coverage now includes:
  - client serialization coverage for the new history query params
  - route coverage for filtered history results
  - persistence coverage proving the same filters survive an app restart
  - browser coverage for `Tag + Status + Attachments + Feedback` together

Useful continuity notes for future sessions:

- history `tag` filtering is based on the app catalog tag set, not custom conversation metadata
  - use `app_policy_watch` when you need a deterministic `policy` tag in browser tests
- persistent history intentionally filters in memory after the SQL read
  - SQL prefilters by `user_id`, `appId`, `groupId`, and `status`
  - final `limit` is applied after `tag/attachment/feedback/query` matching so the returned page stays dense
- if the top-level `npm test` wrapper is too quiet during persistence, split it manually
  - `npm run test:unit`
  - `npm run test:persistence`
  - keep those sequential; do not overlap them with Playwright on the shared Postgres DB
- the browser filter assertion should check both the active-filter badge and the conversation metadata chips
  - `4 active filters`
  - `1 attachments`
  - `Feedback +1 / -0`

## P2-B1 Artifact Contract Freeze

Completed `P2-B1` by freezing a shared artifact DTO family and threading it through the existing chat, transcript, and run boundaries.

Implemented in this round:

- shared contracts now include:
  - `WorkspaceArtifact`
  - `WorkspaceArtifactSummary`
  - artifact `kind`, `source`, `status`, and payload variants for `markdown/text/json/table/link`
- gateway runtime now includes:
  - blocking chat responses that return `choices[0].message.artifacts`
  - streaming final chunks that return `artifacts`
  - persisted assistant transcript messages that keep artifact summaries
  - workspace run detail that exposes full artifacts through `run.artifacts`
- current bootstrap artifact behavior:
  - each assistant completion emits one draft markdown artifact derived from the assistant response body
  - `source = assistant_response`
  - `status = draft`
  - the full artifact currently lives in `runs.outputs.artifacts`
  - transcript messages only store the summary projection
- test coverage now includes:
  - client parsing for blocking and streaming artifact payloads
  - route coverage for blocking completions, streaming final chunks, and run detail
  - persistence coverage that proves artifacts survive restart on both the conversation and run surfaces

Useful continuity notes for future sessions:

- `P2-B1` intentionally does not create a dedicated artifact table
  - artifact storage is still piggybacking on `runs.outputs` and `conversations.inputs.messageHistory`
  - `P2-B2` should introduce the normalized persistence layer and then keep the DTO stable
- transcript messages should keep summaries only
  - full artifact payloads belong on run detail and chat gateway responses
  - this avoids bloating normal conversation reloads before preview/download routes exist
- the final SSE chunk is the artifact handoff point for streaming
  - earlier delta chunks remain content-only
  - browser/UI consumers should bind artifacts from the terminal chunk, then let the conversation refresh catch up
