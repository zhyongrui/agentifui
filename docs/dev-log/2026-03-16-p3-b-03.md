## Continuity Notes

- placeholder runtime tool execution is intentionally opt-in
  - the runtime only synthesizes tool activity when the request explicitly carries `tools`, or when a function `tool_choice` resolves to an enabled registry entry
  - ordinary workspace chat requests without tool metadata still follow the previous no-tool path
- the persisted transcript shape is now:
  - prior request messages from `body.messages`
  - runtime-generated assistant tool-request message with `toolCalls`
  - runtime-generated `tool` message(s) with `toolCallId` and `toolName`
  - final assistant message
- `buildPersistedHistory()` is now the shared write point for this ordering
  - future slices that add latency, retries, approval gates, or richer replay should extend that helper instead of bypassing it
- browser streaming now accepts `delta.tool_calls`, but the current UI only renders a lightweight list
  - richer replay/transcript summaries are still the next dedicated slice `P3-B-09`
- structured run replay data is currently stored under generic `run.outputs.toolCalls` and `run.outputs.toolResults`
  - `P3-B-04` should normalize this into a stricter run-boundary model with latency and attempt metadata
