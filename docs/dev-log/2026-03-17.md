## P3-C-07 Shared metadata conflict handling

Completed `P3-C-07` by adding optimistic concurrency to conversation metadata updates.

Shipped behavior:

- `WorkspaceConversationUpdateRequest` now accepts `expectedUpdatedAt`
- owner and shared-editor update routes both enforce `expectedUpdatedAt` when provided
- stale shared editor updates now return `409 WORKSPACE_ACTION_CONFLICT`
- conflict payload includes `currentUpdatedAt`, `currentTitle`, `currentStatus`, and `currentPinned`
- shared transcript editor UI now sends `expectedUpdatedAt`, refreshes the shared payload after a conflict, and shows a retry message
- conversation metadata update timestamps are now monotonic, so back-to-back updates in the same millisecond still produce a detectable version change

Core files:

- `packages/shared/src/apps/contracts.ts`
- `apps/gateway/src/services/workspace-service.ts`
- `apps/gateway/src/services/persistent-workspace-service.ts`
- `apps/gateway/src/routes/workspace.ts`
- `apps/web/src/app/(main)/chat/shared/[shareId]/page.tsx`
- `apps/web/src/lib/apps-client.test.ts`
- `apps/gateway/src/routes/workspace.test.ts`

Continuity notes:

- if a client wants conflict detection, it must send the last seen `conversation.updatedAt` as `expectedUpdatedAt`
- conflict detection is now safe against same-millisecond updates because owner/shared metadata writes force `updatedAt` to move forward monotonically
- the shared editor UI refreshes the entire shared conversation payload after `WORKSPACE_ACTION_CONFLICT`; reuse that pattern if later surfaces add inline collaborative editors
