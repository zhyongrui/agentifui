## R12-A Attachment Mainline

Completed `R12-A / S2` by turning workspace file uploads into a real conversation-bound capability instead of a placeholder backlog item.

Implemented in this round:

- shared contracts now include:
  - attachment metadata on workspace conversation messages
  - upload request / response DTOs
  - attachment size and MIME allow-list constants
- database and persistence now include:
  - new `workspace_uploaded_files` table
  - migration `0007_r12_a_workspace_uploads.sql`
  - persistent workspace file lookup bound to `conversation_id` and `user_id`
- gateway now includes:
  - local filesystem storage abstraction in `workspace-file-storage.ts`
  - `POST /workspace/conversations/:conversationId/uploads`
  - chat gateway file reference validation for `local_file`
  - attachment metadata persisted into conversation transcript and run inputs
- web now includes:
  - composer attachment picker on `/chat/[conversationId]`
  - upload-before-send flow using the workspace upload route
  - transcript rendering for uploaded attachments
  - run replay rendering for stored attachment metadata
- browser regression now includes:
  - upload `brief.txt`
  - send completion with the uploaded attachment
  - verify transcript and replay both show `Attached files`
  - keep the existing `Stop response` verification in the same flow

Useful continuity notes for future sessions:

- runtime package exports use `dist`, while TypeScript types come from `src`
  - after adding new runtime value exports to `packages/shared` or `packages/db`, `npm run type-check` can still pass while runtime values are `undefined`
  - the minimal fix is to rebuild the affected workspace packages before route tests or ad-hoc runtime smoke:
    - `npm run build --workspace @agentifui/shared`
    - `npm run build --workspace @agentifui/db`
- the new upload route is intentionally conversation-scoped:
  - `POST /workspace/conversations/:conversationId/uploads`
  - this keeps attachment authorization simple for Phase 1 because file ownership and replay both ride on the existing conversation boundary
- the current upload implementation is JSON + base64, not multipart
  - this was chosen to avoid adding a new Fastify multipart layer while keeping the `/api/gateway/*` proxy unchanged
  - if uploads need to grow beyond the current Phase 1 limit, `R13-B` or later should revisit multipart/object storage presign flows
- the Playwright attachment regression uses the stable DOM hook `#chat-attachment`
  - it uploads `Buffer.from('Policy attachment')`
  - it waits for `/api/gateway/workspace/conversations/:id/uploads`
  - it expects the transcript chip and replay section to both surface `brief.txt`
- public access/browser testing context from the prior round still applies
  - temporary `cloudflared tunnel --url http://127.0.0.1:3112 --no-autoupdate`
  - stable `80/443` reverse-proxy hardening remains deferred to `R13-B`

## R12-B Conversation Sharing

Completed `R12-B / S2` by turning conversation sharing from a plan item into a real login-protected, group-scoped read-only surface.

Implemented in this round:

- shared contracts now include:
  - conversation share DTOs, create/list/read responses and `read_only` access mode
  - share lifecycle status values `active` / `revoked`
  - audit actions for share create, access and revoke
- database and persistence now include:
  - new `workspace_conversation_shares` table
  - migration `0008_r12_b_conversation_shares.sql`
  - owner-only share create/list/revoke queries plus shared transcript lookup by `share_id`
- gateway now includes:
  - `GET /workspace/conversations/:conversationId/shares`
  - `POST /workspace/conversations/:conversationId/shares`
  - `DELETE /workspace/conversations/:conversationId/shares/:shareId`
  - `GET /workspace/shares/:shareId`
  - audit writes for `workspace.conversation_share.created`, `.accessed`, `.revoked`
- web now includes:
  - a share management panel on `/chat/[conversationId]`
  - create/revoke controls for group-scoped read-only shares
  - `Open shared view` links that target `/chat/shared/[shareId]`
  - a dedicated read-only shared transcript page with attachments but no composer
- browser regression now includes:
  - owner creates a conversation in `Research Lab`
  - owner creates a read-only share for `grp_research`
  - a different member logs in and opens the shared URL
  - the shared transcript loads and the message composer is absent

Useful continuity notes for future sessions:

- shared conversation access is not public/anonymous
  - the consumer must still hold a valid workspace session
  - `GET /workspace/shares/:shareId` then enforces active membership in the shared target group
  - a revoked share or a user outside the group will get a forbidden/not-found response instead of a degraded partial view
- share URLs are stable app routes, not raw gateway links
  - the canonical user-facing path is `/chat/shared/:shareId`
  - the Next.js page calls the gateway through the existing `/api/gateway/*` proxy, so no extra public routing is needed for Phase 1
- share management is intentionally owner-scoped in this phase
  - only the conversation owner can list/create/revoke shares
  - readers only get the shared transcript surface, not share metadata editing controls
- the shared transcript page reuses the same bubble styling as the main chat surface
  - when asserting transcript content in Playwright, scope to `.chat-bubble.user p` or another bubble-local locator
  - plain `getByText(...)` can hit both the explanatory copy and the transcript body once prompts resemble page text
- the earlier public browser-access note still applies unchanged
  - temporary manual access can still be provided with `cloudflared tunnel --url http://127.0.0.1:3112 --no-autoupdate`
  - stable `80/443` ingress hardening remains deferred to `R13-B`
