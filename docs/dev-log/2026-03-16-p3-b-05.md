## Continuity Notes

- approval-gated tool context is currently transported in `WorkspaceHitlStep.metadata`
  - keys in use:
    - `kind = tool_approval`
    - `toolCallId`
    - `toolName`
    - `toolArguments`
    - optional `policyTag`
- do not treat `outputs.toolCalls` alone as proof that a tool executed
  - replay fallback now requires a matching tool-result payload before deriving a legacy execution row
- approval/reject/cancel currently resolve onto the same run instead of opening a new run
  - this is intentional so replay preserves the full request -> approval -> result chain in one place
- browser/UI integrations must reload the conversation after responding to a pending action
  - otherwise the newly appended tool/result assistant messages will not appear
- the seeded approval-required test path is `tenant.access.review`
  - use it for API/browser smoke until broader real tool integrations land
