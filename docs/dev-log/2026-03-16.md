## P3-B-05 approval-required tool execution paths

This follow-up reused the existing HITL boundary to gate selected tool calls behind an explicit approval step.

Implemented in this round:

- added `WorkspaceHitlStep.metadata`
  - tool-approval steps now carry:
    - `kind = tool_approval`
    - `toolCallId`
    - `toolName`
    - serialized tool arguments
    - optional `policyTag`
- marked `tenant.access.review` as the first approval-required seeded tool
- updated the placeholder runtime
  - explicit `tool_choice` requests for approval-gated tools now return:
    - assistant `tool_calls`
    - a pending approval step
    - no immediate `toolResults` / `toolExecutions`
- updated pending-action response handling in both memory and persistent workspace services
  - approving/rejecting/cancelling a tool-approval step now appends:
    - a persisted `tool` transcript message
    - a persisted assistant follow-up message
    - `outputs.toolResults`
    - `outputs.toolExecutions`
    - an additional `output_recorded` timeline event
- updated the chat page pending-action response flow
  - after a response, the page now reloads the conversation and selected run from the server instead of only patching local `pendingActions`
- tightened legacy replay fallback
  - derived tool executions now ignore `toolCalls` that have no matching result payload
  - this prevents approval-pending runs from incorrectly rendering as already-succeeded executions

Continuity notes:

- the tool-approval transport is currently `WorkspaceHitlStep.metadata`
  - future runtimes should reuse the same metadata keys instead of inventing a second approval envelope
- the approval resolution currently mutates the same run that requested the tool
  - transcript replay therefore shows:
    - the initial assistant tool-call request
    - the later tool result
    - the later assistant approval-resolution follow-up
- if the frontend only patches `pendingActions` locally, replay goes stale
  - the required post-response behavior is now:
    - reload conversation
    - reload run detail
    - prefer the returned `runId`
- `tenant.access.review` is now the seeded approval-required path for tests
  - use that tool for browser/API validation until richer real tools exist
