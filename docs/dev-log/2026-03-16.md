Implemented in this round:

- expanded the shared failure taxonomy in:
  - `packages/shared/src/apps/contracts.ts`
  - added tool-stage codes:
    - `tool_timeout`
    - `tool_provider_error`
    - `tool_approval_rejected`
    - `tool_approval_cancelled`
    - `tool_approval_expired`
  - added tool-stage stages:
    - `tool_execution`
    - `tool_approval`
  - `WorkspaceRunToolExecution` now supports a structured `failure` field
- added tool-failure derivation helpers in:
  - `apps/gateway/src/services/workspace-run-failure.ts`
  - failed tool executions can now derive structured failure objects from:
    - explicit persisted `failure`
    - or legacy `metadata.failureReason`
- wired the new failure payload into:
  - `apps/gateway/src/routes/chat.ts`
  - `apps/gateway/src/services/workspace-tool-approval.ts`
  - `apps/gateway/src/services/workspace-service.ts`
  - `apps/gateway/src/services/persistent-workspace-service.ts`
  - audit payloads now also include `execution.failure`
- updated replay rendering in:
  - `apps/web/src/components/workspace-tool-summary.tsx`
  - replay cards now show:
    - failure code
    - failure stage
    - failure message
    - retryability

Continuity notes:

- do not overload `run.failure` to describe every failed tool attempt
  - a run can still succeed while some tool attempts fail
  - those failures now belong on `toolExecutions[*].failure`
- new persisted tool execution rows should write both:
  - `metadata.failureReason` for old consumers
  - structured `failure` for new readers
- readers remain backward compatible
  - if `toolExecutions[*].failure` is missing, they derive it from legacy metadata and result content

## P3-C-01 Conversation presence/session model

Completed `P3-C-01` by adding the first presence/session boundary for multiple viewers on the same owner conversation surface.

Implemented in this round:

- extended shared contracts in:
  - `packages/shared/src/apps/contracts.ts`
  - added:
    - `WorkspaceConversationPresenceEntry`
    - `WorkspaceConversationPresence`
    - `WorkspaceConversationPresenceUpdateRequest`
    - `WorkspaceConversationPresenceResponse`
- added shared presence helpers in:
  - `apps/gateway/src/services/workspace-presence.ts`
  - the current model is:
    - process-local
    - ttl-based
    - session keyed
    - per-conversation
- added owner routes in:
  - `apps/gateway/src/routes/workspace.ts`
  - new endpoints:
    - `GET /workspace/conversations/:conversationId/presence`
    - `PUT /workspace/conversations/:conversationId/presence`
- wired both memory and persistent workspace services:
  - `apps/gateway/src/services/workspace-service.ts`
  - `apps/gateway/src/services/persistent-workspace-service.ts`
- added same-origin web client helpers:
  - `apps/web/src/lib/apps-client.ts`
  - `fetchWorkspaceConversationPresence(...)`
  - `updateWorkspaceConversationPresence(...)`

Continuity notes:

- current presence is intentionally ephemeral
  - it is process-local and ttl-pruned
  - it does not survive gateway restart
  - this is acceptable for collaboration presence, but not for durable comments or review state
- `sessionId` is client-provided
  - the server does not derive it from auth session token
  - future browser work should keep one stable client session id per tab/surface
- `P3-C-01` only covers owner conversation presence
  - shared transcript presence, collaborator chips, and multi-user UI are later tasks
