# 2026-03-16 CI E2E Stabilization

## Summary

- investigated the current GitHub Actions failures and confirmed they were isolated to the `e2e` job
- reproduced the browser failures locally and fixed two separate issues in the e2e harness/assertions
- revalidated the repo with `type-check`, `build`, `test:unit`, and a full green `npm run test:e2e`

## What Failed

- GitHub Actions run `23126078415` on push `d378b4a` had:
  - `typecheck` green
  - `build` green
  - `db-validate` green
  - `unit` green
  - only `e2e` red
- local reproduction showed two real browser-suite failure modes:
  - `tests/e2e/knowledge-flows.spec.ts`
    - the replay panel can contain the same source title more than once
    - the old locator assumed strict text uniqueness and failed when retrieval-backed replay rendered duplicates
  - `tests/e2e/phase1-flows.spec.ts`
    - later `/apps` scenarios could fail before launch because `Policy Watch` showed `配额不足`
    - root cause was accumulated tenant/group quota usage across the shared e2e suite state, not a product authorization or group-switch regression

## Implementation

- updated `tests/e2e/knowledge-flows.spec.ts`
  - the replay assertion now scopes to the run-history panel and tolerates duplicate source titles via a filtered locator with `.first()`
- updated `scripts/run-e2e.mjs`
  - added free-port selection so the wrapper no longer depends on `3111/4111` being idle
  - added quota seeding for the shared e2e tenant and key groups:
    - tenant `dev-tenant` or `GATEWAY_DEFAULT_TENANT_ID`
    - groups `grp_research`, `grp_security`, `grp_product`
  - the quota seed happens after gateway health succeeds
    - doing it earlier can fail the `workspace_quota_limits.tenant_id` foreign key because the tenant bootstrap rows are not guaranteed to exist yet

## Continuity Notes

- if CI shows broad red, first split it by job
  - for the current failure cluster the product code was not the problem; only `e2e` was failing
- `npm run test:e2e` now auto-picks free ports
  - do not hard-code a browser base URL assumption around `3111`
  - for raw Playwright you still need to provide `PLAYWRIGHT_BASE_URL`
- the browser suite shares mutable quota state
  - long suites can consume launch/run budget and break later `/apps` launch tests
  - if a card flips from `打开应用` to `配额不足` late in the run, inspect quota accumulation before changing product logic
- replay/source panels can legally repeat the same title
  - avoid strict unique-text locators for retrieval-backed source assertions
- this host still prefers sequential heavy test phases
  - do not overlap persistence, full Playwright, and other DB-heavy flows on the shared local Postgres instance
